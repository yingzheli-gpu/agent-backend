**关键特性**:
- 医疗场景先总结旧消息（保留症状、证型、用药反馈），再裁剪
- 主图保留最近10条消息，诊断子图保留最近8条
- 使用 `TCMSummarizer` 生成医疗专用摘要

## 关键设计决策

### ADR-08: 总结-裁剪策略适用于所有医疗场景

**决策**: 主图和诊断子图都启用总结-裁剪，养生子图禁用。

**原因**:
1. 医疗对话中早期信息（症状、证型、用药反馈）对后续诊断至关重要
2. 直接裁剪会丢失关键医疗信息，导致诊断错误
3. 总结可以保留关键信息同时减少 token 消耗
4. 养生对话通常较短，不需要总结

### ADR-09: 子图不重复加载 Memory，但需要独立的 Focus 和 Learning

**决策**: 子图继承主图的 `memory_context`，但有独立的 Focus 和 Learning 中间件。

**原因**:
1. Memory (用户画像、历史情景) 是全局稳定数据，不需要重复检索
2. Focus 压缩策略因子图特性不同（主图无工具，诊断子图工具密集）
3. Learning 经验需要按 source 过滤（主图仅意图学习，子图包含工具学习）

### ADR-10: 子图通过 handler 传递 conversation_id/user_id

**决策**: 不修改子图 state schema，通过 handler 函数传递中间件所需字段。

**原因**:
1. 最小侵入性，不破坏现有 state 定义
2. 中间件可以处理任何 dict-like state
3. 避免修改所有子图节点的签名
