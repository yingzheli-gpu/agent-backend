# 子图工具选择学习机制优化方案

## 1. 背景与目标

本文只回答一个核心问题：

> 在 `final_agent` 架构里，“工具选择学习机制”到底应该放在主图，还是应该放在诊断子图、养生子图这类子图里？现有两份优化文档是否已经把这件事设计清楚？

结论先行：

1. **工具选择学习的归属属于子图域能力，而不是主图路由能力。**
2. **实现上不建议每个子图各自维护一套 learner。** 更合理的方案是：
   - 主图保留统一的学习基础设施（`SelfLearner`、`LearningMiddleware`、存储层、聚合层）；
   - 各子图负责定义“何时触发工具选择学习、学什么、如何消费”；
   - 通过 `source="diagnose_subgraph"`、`source="wellness_subgraph"` 做隔离与复用。
3. **你提到的两份优化文档都没有把“子图工具选择学习闭环”设计完整。** 它们更多是在设计“上下文透传”和“学习结果注入”，而不是“子图内部如何产生日志、沉淀经验、回放经验”。

因此，这份文档的目标不是写代码，而是先把 **学习机制的分层边界、诊断/养生两类子图的差异、以及后续真正该补的文档与实现接口** 定义清楚。

## 2. 当前架构事实（基于现有代码）

### 2.1 主图的职责已经比较清晰

从 `backend/app/src/final_agent/__init__.py` 与 `backend/app/src/final_agent/builder.py` 可以看到，当前主图设计是：

- 主图负责请求级通用能力：安全、长期记忆、上下文压缩、学习、脱敏；
- 子图负责业务执行：诊断、养生、药材、方剂等；
- 主图中已经接入 `MemoryMiddleware` 与 `LearningMiddleware`。

也就是说，**学习基础设施目前已经放在主图层**，这没有问题；问题不在“有没有 learner”，而在“子图是否真正参与学习闭环”。

### 2.2 学习数据结构已经为“子图工具学习”预留了位子

从 `backend/app/src/final_agent/states.py` 可见，当前状态层已经定义了：

- `ToolSelectionCorrectionRecord`：子图工具选择纠错记录；
- `ThreadLearningContext`：当前线程学习快照；
- `CrossThreadLearningContext`：跨线程学习上下文；
- `tool_learning`、`correction_learning`：明确属于子图学习，不属于主图意图学习。

从 `backend/app/src/final_agent/learning/events.py` 可见，也已经有：

- `LearningEventType.TOOL_SELECTION_CORRECTION`；
- `ToolSelectionCorrection` 数据结构；
- `subgraph` / `source` 字段，支持按子图隔离学习事件。

这说明：

**从状态模型和事件模型上，系统已经承认“工具选择学习属于子图维度”。**

### 2.3 LearningMiddleware 的设计方向是“共享服务 + source 过滤”

从 `backend/app/src/final_agent/middleware/learning.py` 可见：

- `LearningMiddleware` 支持 `source` 参数；
- `source` 明确区分：`main_graph`、`diagnose_subgraph`、`wellness_subgraph`；
- `before_model` 中已经有不同 source 的加载策略：
  - `main_graph`：只加载意图学习；
  - `diagnose_subgraph`：应加载完整学习经验；
  - `wellness_subgraph`：应加载养生专属策略；
- `_load_cross_thread_learning()` 中也明确写了：
  - `wellness_subgraph` 不加载鉴别规则与误诊模式；
  - `main_graph` 不加载具体策略。

所以从中间件设计意图看，当前架构实际上已经倾向于：

**工具选择学习不在主图“做决策”，而是在共享学习服务中按子图视角加载和过滤。**

### 2.4 真正缺的不是“设计名词”，而是“子图闭环接线”

虽然上面的数据结构和中间件方向都存在，但从当前业务子图代码看，接线并没有真正完成。

此外，共享学习层本身也还有一个值得注意的接口错位：

- `backend/app/src/final_agent/middleware/learning.py` 在 `before_model()` 中以 `source=self.source` 调用 `self.learner.get_thread_learning_snapshot(...)`；
- 但 `backend/app/src/final_agent/learning/learner.py` 当前 `get_thread_learning_snapshot()` 的签名只有 `session_id`，并没有 `source` 参数。

这说明当前代码状态并不是“设计已经完整，只差子图透传字段”，而是：

**共享学习层与子图学习层之间，仍然存在接口契约尚未收敛的问题。**

#### 诊断子图现状

从以下文件可见：

- `backend/app/src/final_agent/components/diagnose/handlers.py`
- `backend/app/src/final_agent/components/diagnose/states.py`
- `backend/app/src/final_agent/components/diagnose/builder.py`
- `backend/app/src/final_agent/components/diagnose/nodes/simple/simple_diagnosis.py`

- `backend/app/src/final_agent/components/diagnose/nodes/complex/complex_diagnosis.py`

可以确认：

1. 诊断子图输入状态目前还没有 `memory_context`、`cross_thread_learning`、`thread_learning_context`；
2. `handle_diagnose_query()` 目前也没有把这些学习上下文传进子图；

6. `complex_diagnosis` 虽然是最需要工具学习的地方，但当前只是新建一个 DeepAgent 并传入单条 prompt，没有把历史学习经验注入进去。

换句话说：

**诊断子图在“应该学习工具选择”这件事上，概念是有的，结构体也有，但运行闭环没有打通。**

#### 养生子图现状

从以下文件可见：

- `backend/app/src/final_agent/components/wellness/states.py`
- `backend/app/src/final_agent/components/wellness/handlers.py`
- `backend/app/src/final_agent/components/wellness/builder.py`
- `backend/app/src/final_agent/tools/registry.py`

可以确认：

1. 养生子图输入状态更轻，只包含 `query`、`user_profile`、`sub_type`、`llm_config`；
2. 当前养生子图没有显式工具编排链；
3. `AGENT_TOOL_MAPPING` 里 `wellness` 当前是空工具列表；
4. 养生处理节点本质上仍是直接 LLM 回答，而不是“多工具选择执行”。

这意味着：

**从当前代码现实出发，养生子图还没有进入“真正的工具选择学习阶段”，它更像是“路由/策略选择学习”，而不是“工具调用顺序学习”。**

## 3. 对现有两份优化文档的覆盖度判断

本节直接回答你的问题：

> 现有优化文档有设计么？

答案是：**有部分设计，但没有完整设计到“子图工具选择学习机制”。**

### 3.1 `长期记忆整合计划.md` 的覆盖范围

`backend/app/src/final_agent/docs/长期记忆整合计划.md` 的重点是：

- 主图到诊断子图的数据透传；
- `memory_context`、`cross_thread_learning`、`thread_learning_context` 如何进入诊断子图；
- simple / moderate / complex 三条诊断路径如何消费长期记忆与学习上下文；
- complex 路径如何通过 prompt/tool 使用记忆与学习规则。

它解决的是：

- **子图“看见什么学习结果”**；
- **长期记忆如何进入诊断流程**；
- **复杂诊断如何使用已有经验辅助决策**。

但它没有完整定义：

1. 子图内部何时判定“工具选择错误”；
2. 诊断子图如何记录工具选择事件；
3. 工具选择经验如何从线程级汇总成跨线程规则；
4. 养生子图是否要有工具学习，若有与诊断子图有何不同。

因此，这份文档本质上是：

**“学习结果注入设计” > “子图工具学习生产设计”。**

### 3.2 `诊断子图和养生子图的优化.md` 的覆盖范围

`backend/app/src/final_agent/docs/诊断子图和养生子图的优化.md` 的重点是：

- 诊断与养生子图的上下文闭环；
- State 扩展；
- 主图到子图的参数透传；
- complex diagnosis 的 prompt/tool 增强；
- 养生子图如何继承上下文。

它确实比上一份更接近“子图学习”，尤其诊断部分已经提到：

- 诊断子图需要独立学习经验；
- 诊断子图可能包含 `tool_learning`、`correction_learning`；
- 养生子图更偏 `effective_strategies`。

但问题在于，这份文档仍然主要在讲：

- 子图如何接收上下文；
- prompt 如何注入学习经验；
- 需要新增哪些状态字段。

它仍然没有完整覆盖：

1. **学习触发点**：什么算工具选错，什么算策略选错；
2. **学习单元**：记录的是单次错误、阶段策略还是完整工具序列；
3. **升级路径**：线程级纠错何时升级为跨线程规则；
4. **诊断/养生差异化规范**：诊断是强工具学习，养生是弱工具学习还是先不学；
5. **主图与子图之间的职责边界**：谁负责采集，谁负责聚合，谁负责消费。

因此，这份文档可以视为：

**“子图上下文闭环设计” 已有，**
**“子图工具选择学习闭环设计” 仍未成型。**

### 3.3 一个关键补充：`学习机制.md` 其实已经给出概念答案

虽然你这次主要问的是两份优化文档，但需要说明的是：

`backend/app/src/final_agent/docs/学习机制.md` 在概念层已经明确把：

- 意图识别学习归到主图；
- 工具选择准确率、症状理解准确率、辨证推理准确率、追问策略准确率归到子图。

尤其其中“工具选择准确率”这一节，已经明确说明：

- 这是 **子图学习维度**；
- 包含单线程学习与跨线程学习；
- 目标是形成工具调用时机规则、工具选择决策树、信息完整性检查清单。

所以你的疑问在概念层其实已经被回答过：

**是的，工具选择学习机制应该属于子图学习。**

真正缺的是：

**如何把这个概念，落实到“诊断子图”和“养生子图”的差异化技术设计里。**

## 4. 学习机制的正确分层

为避免后续设计混乱，建议把学习能力拆成四层。

### 4.1 第 1 层：主图学习（全局入口学习）

负责内容：

- 意图识别纠偏；
- 路由误判纠偏；
- 用户续问/省略句理解纠偏。

特点：

- 关心“该去哪个子图”；
- 不关心“子图内部用了哪个工具”。

结论：

**主图不能承担工具选择学习。**

因为主图不知道诊断子图里的 `collect_info -> moderate -> complex` 具体为什么失败，也不知道养生子图里某次 web 增强是否必要。

### 4.2 第 2 层：子图工具选择学习（域内执行学习）

负责内容：

- 某个子图内部是否应该调用工具；
- 应该先调哪个工具，再调哪个工具；
- 什么信息不足时不该过早进入检索/诊断/综合；
- 哪些工具序列在某类病例/某类养生任务上更有效。

特点：

- 这是 **领域相关** 的；
- 必须依赖子图内部状态与阶段信息；
- 同样叫“工具学习”，诊断和养生的定义并不相同。

结论：

**工具选择学习必须由子图定义触发语义。**

### 4.3 第 3 层：共享学习服务（统一存储与聚合）

负责内容：

- 统一保存 `LearningEvent`；
- 保存 thread-level 学习上下文；
- 汇总 cross-thread 规则、误诊模式、高效策略；
- 按 `source/subgraph` 做检索和过滤。

特点：

- 它是基础设施，不是业务决策层；
- 不定义“什么算选错工具”，只接收子图给出的事件与标签。

结论：

**learner 可以共享，但学习语义不能共享。**

### 4.4 第 4 层：长期记忆系统（用户记忆，不是策略记忆）

负责内容：

- 用户画像；
- 历史诊断摘要；
- 历史疗效反馈；
- 稳定实体与语义关系。

注意：

长期记忆不是工具选择学习本身，但它会影响工具选择学习：

- 某用户过去多次出现复杂兼证，那么诊断子图应更早使用复杂路径；
- 某用户已有稳定体质画像，那么养生子图可能不必重复走体质判定工具链。

因此：

**长期记忆是工具选择学习的重要条件输入，但不是学习归属。**

## 5. 核心设计结论：工具选择学习应“归属子图、托管共享 learner”

这是本方案最重要的架构判断。

### 5.1 不建议的错误设计

#### 方案 A：全部放在主图学

问题：

- 主图只能看到“最后成败”，看不到子图内部阶段与工具原因；
- 会把诊断子图和养生子图的学习规则混在一起；
- 生成的规则往往过于粗糙，例如“某类问题少用 web search”，但不知道是诊断还是养生场景。

#### 方案 B：每个子图自建一套 learner

问题：

- 存储碎片化；
- 难以统一做跨线程聚合；
- 无法共用状态结构与评估指标；
- 后续新增 herb/prescription 子图时会不断复制架构。

### 5.2 推荐的目标设计

推荐架构：

- **主图**：保留统一 `SelfLearner` / `LearningMiddleware` / `storage`；
- **子图**：负责记录本域学习事件，并定义本域的学习内容；
- **事件层**：所有子图都写统一 `LearningEvent`，但必须带 `source/subgraph`；
- **消费层**：子图进入前只加载自己的学习视图，不混用其他子图规则。

一句话概括就是：

> **学习服务共享，学习语义分域，学习经验按子图隔离。**

## 6. 诊断子图的工具选择学习设计

诊断子图是当前最需要做“工具选择学习”的部分。

### 6.1 诊断子图为什么必须做

原因有三：

1. 它内部天然存在多阶段决策：收集信息、评估复杂度、simple/moderate/complex 路由；
2. moderate 与 complex 路径都涉及显式工具或工具代理链；
3. 工具选错的代价大，直接影响误诊、漏诊、追问冗余与 token 成本。

### 6.2 诊断子图要学的不是“某个工具好不好”，而是“四类规则”

#### A. 时机规则

例如：

- 信息不足时不应过早进入 moderate/complex；
- 缺少寒热/二便/舌脉等关键项时，不应直接做综合辨证；
- 同一轮里先补信息还是先检索知识图谱。

#### B. 顺序规则

例如：

- 先 `collect_info` 再 `syndrome search`；
- 先理论检索再方剂建议；
- 先鉴别规则校验再输出结论。

#### C. 升降级规则

例如：

- 哪些病例应从 simple 升到 moderate；
- 哪些 moderate 病例应该直接进入 complex DeepAgent；
- 哪些复杂病例实际上可以在 moderate 结束，不必触发昂贵链路。

#### D. 终止规则

例如：

- 当前信息已足够，继续搜索只会增加噪声；
- 当前已有明确高风险误诊模式，必须追加鉴别；
- 当前工具结果相互冲突，先进入验证环节而不是继续查。

### 6.3 诊断子图学习事件的标准化建议

诊断子图的工具学习事件建议统一落到以下语义：

- `wrong_tool`：选错了哪类工具/节点；
- `correct_tool`：下一次应选什么；
- `reason`：为什么错；
- `missing_info`：当时缺了什么；
- `hint`：可复用的决策提示；
- `stage`：发生在收集/评估/moderate/complex 哪个阶段；
- `case_signature`：本次病例的关键特征摘要；
- `tool_sequence`：本轮已经走过的工具序列；
- `outcome`：导致了误诊、冗余追问、空检索、重复检索还是高成本低收益。

这里要强调：

**诊断子图要学习的是“决策过程”，不是只记录一个工具名字。**

### 6.4 诊断子图学习闭环

建议闭环如下：

1. 子图入口：加载 `source="diagnose_subgraph"` 的 thread/cross-thread 学习视图；
2. 阶段节点：在 `collect_info`、`route_by_complexity`、moderate planner、complex agent planner 处消费学习经验；
3. 运行中：遇到错误工具选择、重复查询、无效查询时记录事件；
4. 子图退出：汇总当前病例的工具序列、失败点、有效策略；
5. 聚合层：把高频 thread-level 纠错升级为跨线程诊断策略与规则。

### 6.5 诊断子图最终应得到三类可回放资产

#### 1. 线程级即时纠偏

用于同一次问诊内快速纠正：

- 本轮不要再重复问寒热；
- 当前病例缺少舌象，先别进入复杂综合；
- 此类患者先看定位症状再查方剂。

#### 2. 跨线程工具策略

用于同类病例复用：

- 某类复杂兼证优先走 `moderate -> complex validation`；
- 某类证型混淆先查鉴别规则，再查相似医案；
- 某类检索词组合更高效。

#### 3. 成功工具序列模板

这是诊断子图最容易被忽略、但价值很高的一类“程序性记忆”：

- 对什么症状组合，什么工具顺序最省轮次；
- 对什么复杂度区间，什么路径最稳。

## 7. 养生子图的工具选择学习设计

养生子图不能简单复制诊断子图方案，必须做弱化与分层。

### 7.1 现实判断：当前养生子图还不是“重工具子图”

基于当前代码，养生子图主要是：

- 根据 `sub_type` 做路由；
- 直接构造 prompt 给 LLM 回答；
- 没有显式工具注册和工具编排链。

因此，现阶段如果直接照搬“工具选择学习”，会出现设计过度。

### 7.2 养生子图当前更适合做“策略选择学习”

养生子图现阶段更适合学习的是：

- 是否应该继续停留在 wellness，而不是升级到 diagnosis；
- 是否应该走季节养生、体质调理还是通用养生；
- 是否需要 L2 增强（例如外部检索或更长上下文），而不是轻量直接答；
- 哪类用户画像应优先引用历史偏好和体质信息。

换句话说：

**养生子图当前学习的重点不是“选哪把工具”，而是“选哪种处理策略”。**

### 7.3 养生子图的学习应分两阶段建设

#### 阶段 1：策略学习（当前就该设计）

学习对象：

- 路由策略；
- 上下文使用策略；
- 是否需要外部增强；
- 何时建议转问诊子图。

输出资产：

- 高满意度养生方案模板；
- 用户画像到方案的映射偏好；
- 易误导的养生问题升级规则。

#### 阶段 2：工具学习（等养生子图真正引入工具后再落地）

未来当养生子图显式引入以下能力时，再开启完整工具学习：

- 季节知识检索；
- 体质库查询；
- 食疗/穴位知识图谱；
- 外部养生资讯检索；
- 历史建议回查工具。

此时养生子图就可以与诊断子图共用相同事件模型，但规则语义会更轻：

- 是否需要检索；
- 应优先检索哪类养生知识；
- 是否应先确认体质，再给建议；
- 是否应直接提示边界并转诊断子图。

### 7.4 养生子图当前不应强行引入的内容

在现阶段，养生子图**不建议**强行引入以下诊断型资产：

- 诊断鉴别规则；
- 误诊模式库；
- 复杂病例工具序列；
- 诊断成功案例程序性记忆。

因为这些都是诊断域知识，会污染养生子图的决策边界。

## 8. 子图学习与长期记忆的关系

这部分很关键，因为你当前的两份优化文档都在讨论记忆透传。

### 8.1 长期记忆回答“用户是谁，过去发生过什么”

它解决的问题是：

- 用户过去是什么体质；
- 是否有稳定既往史；
- 以前诊断过什么；
- 以前什么建议有效或无效。

### 8.2 子图工具学习回答“这次该怎么做更对”

它解决的问题是：

- 这次应先检索还是先追问；
- 这类问题是否该走复杂路径；
- 这类养生问题是否该做外部增强；
- 哪种工具序列更稳、更省轮次。

### 8.3 两者如何配合

推荐关系是：

- **长期记忆提供条件**：用户背景、历史病程、体质、既往疗效；
- **子图学习提供策略**：面对这样的条件，工具与步骤如何选；
- **跨线程学习提供可泛化规则**：这不是某个用户独有，而是一类问题都适用。

因此：

> 记忆决定“看见谁”，学习决定“怎么做”。

### 8.4 学习资产的读取优先级

为了避免子图在运行时出现规则冲突，建议统一采用下面的读取优先级：

1. **本线程即时纠偏**：优先级最高，用于修正当前会话里刚刚暴露出的错误；
2. **跨线程稳定规则**：作为通用经验，用于补充当前线程没有覆盖到的模式；
3. **长期记忆条件信息**：作为用户背景与历史条件，不直接覆盖学习规则，只参与决策约束。

这个优先级的含义是：

- 如果用户刚在当前线程纠正了某个理解或工具时机，子图必须优先服从当前线程纠偏；
- 如果当前线程没有直接纠偏，再参考跨线程经验，例如高频误诊模式、成功工具序列；
- 长期记忆只回答“这个用户过去是什么情况”，不能反过来压过当前线程纠偏或跨线程规则。

一句话概括：

> **本线程纠偏管当下，跨线程规则管共性，长期记忆管背景。**

## 10. 最终结论

最后把你的问题压缩成一句最明确的话：

### 10.1 关于“工具选择学习机制是在子图中学习的吗？”

**是。语义上，它必须属于子图学习。**

因为只有子图知道：

- 当前阶段是什么；
- 为什么要选这个工具；
- 为什么这个工具在这个阶段选错了；
- 哪类病例/哪类养生问题更适合哪条执行路径。

### 10.2 关于“现有优化文档有设计吗？”

**有部分设计，但没有完整设计。**

- `长期记忆整合计划.md`：重点是记忆与学习结果如何进入诊断子图；
- `诊断子图和养生子图的优化.md`：重点是上下文闭环与子图透传；
- 两者都还没有完整定义“子图工具选择学习”的触发、记录、聚合、回放闭环；
- 真正把“工具选择学习属于子图”讲清楚的，是 `学习机制.md`，但它还停留在概念层。

### 10.3 推荐的架构原则

最终建议采用下面这条原则作为后续所有设计和实现的总原则：

> **工具选择学习归属子图，存储与聚合复用共享 learner，诊断子图做强工具学习，养生子图先做策略学习、后做工具学习。**

这条原则能同时满足：

- 不把诊断和养生规则混在一起；
- 不重复造学习基础设施；
- 能与长期记忆系统自然配合；
- 能兼容你现有 `states.py`、`events.py`、`middleware/learning.py` 的结构。
