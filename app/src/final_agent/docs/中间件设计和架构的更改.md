## 2. 图层级定义

本系统共4层:

| 层级 | 名称 | 职责 | 是否调用工具 | 是否调用LLM推理 |
|------|------|------|-------------|----------------|
| L0 | 主图 Main Graph | 意图识别 + 路由分流 | 否 | 是(分类器) |
| L1 | 养生子图 Wellness Subgraph | 二级路由 + 养生回复生成 | 轻量(查节气等) | 是 |
| L1 | 诊断子图 Diagnose Subgraph | 信息收集 + 复杂度评估 + 辨证 | 是(中/重度) | 是 |
| L2 | DeepSearch Agent | 复杂病例深度辨证 | 重度(KG/向量/古籍/Web) | 是(5个专家子Agent) |

## 3. 记忆类型定义(认知科学三层模型)

基于最新研究(MachineLearningMastery 2025.12, G-Memory 2025.06), 定义三种长期记忆:

### 3.1 Factual Memory (事实记忆)

**存储位置**: PostgreSQL (patients 表)
**内容**: 用户画像、性别、年龄、体质、过敏史、病史
**特点**: 稳定、跨会话不变、结构化

```python
factual = {
    "user_profile": {
        "gender": "女",
        "birth_date": "1991-05-12",
        "constitution": "气郁质",
        "medical_history": ["慢性胃炎"],
        "allergy_info": ["青霉素过敏"],
        "taboo_items": ["辛辣"]
    },
    "source": "database",
    "locations": ["patients.base_profile", "patients.gender", "patients.birth_date"]
}
```

### 3.2 Episodic Memory (情景记忆)

**存储位置**: Mem0 (向量存储 + 图存储)
**内容**: 历史诊断情景摘要、用户反馈、过去的证型/方剂
**特点**: 面向用户回顾("你上次诊断了什么"), 按语义检索

```python
episodic = {
    "summaries": [
        {
            "memory_id": "mem_xxx",
            "summary": "2周前诊断为肝郁脾虚, 建议逍遥散, 用户反馈效果良好",
            "timestamp": "2026-02-24",
            "metadata": {
                "syndrome": "肝郁脾虚",
                "prescription": "逍遥散",
                "task_type": "tcm-diagnose",
                "user_satisfaction": 4.5
            }
        }
    ],
    "source": "mem0"
}
```

### 3.3 Semantic Memory (语义记忆)

**存储位置**: Mem0 Graph (Neo4j 图谱)
**内容**: 实体关系、稳定的知识关联(证型-方剂-药材)
**特点**: 面向知识推理, 图遍历

```python
semantic = {
    "relations": [
        {"source": "肝郁脾虚", "relation": "常用方剂", "target": "逍遥散"},
        {"source": "逍遥散",   "relation": "君药",     "target": "柴胡"}
    ],
    "entities": ["肝郁脾虚", "逍遥散", "柴胡", "白芍"],
    "source": "mem0_graph"
}
```

### 3.4 Procedural Memory (程序记忆 = 学习经验)

**存储位置**: 数据库 (learning 相关表) + SelfLearner 内存
**内容**: 工具调用序列、成功诊断策略、纠错规则、误诊模式
**特点**: 面向 Agent 能力提升("遇到这种情况应该怎么做")

```python
procedural = {
    # 单线程学习 (当前会话)
    "thread_learning": {
        "intent_corrections": [...],     # 意图纠错
        "error_reflections": [...],      # 错误反思
        "effective_strategies": [...]     # 有效策略
    },
    # 跨线程学习 (历史积累)
    "cross_thread_learning": {
        "discriminating_rules": [...],   # 鉴别规则
        "misdiagnosis_patterns": [...],  # 误诊模式
        "effective_strategies": [...],   # 高效策略
        "successful_cases": [...]        # 成功案例 (仅子图使用)
    }
}
```

## 4. 记忆 vs 学习经验的区别

这是核心设计决策, 必须明确:

| 维度 | 记忆 (Memory) | 学习经验 (Learning) |
|------|--------------|-------------------|
| 面向对象 | 用户 | Agent自身 |
| 回答的问题 | "上次诊断了什么?" | "遇到这种情况怎么做?" |
| 存储位置 | Mem0 (向量+图) | 数据库 (结构化表) |
| 检索方式 | 语义相似度检索 | 规则匹配 + Top-N |
| 注入时机 | 主图 before_model | 各层级 before_model |
| 中间件 | MemoryMiddleware | LearningMiddleware |

**关于成功诊断案例的双重存储**:

同一次成功诊断, 会产生两份记录:

1. **情景记忆** (Mem0, 由主图 MemoryMiddleware.after_model 保存):
   - 内容: "用户主诉胁痛腹胀, 诊断为肝郁脾虚, 建议逍遥散"
   - 用途: 下次用户问"我上次诊断结果是什么"时检索到

2. **程序记忆** (数据库, 由子图 LearningMiddleware.after_model 保存):
   - 内容: `{syndrome, complexity_score, tool_sequence, diagnosis_rounds, satisfaction}`
   - 用途: 下次遇到类似复杂度案例时, 参考工具调用序列和诊断策略

**不重复**: 两者存储内容、存储位置、检索方式、使用场景完全不同, 不构成冗余。

## 5. LangGraph 记忆机制说明

### 5.1 Checkpointer vs Store vs Mem0

| 组件 | 作用 | 在本项目中的使用 |
|------|------|-----------------|
| **Checkpointer** | 短期记忆，自动维护对话历史 (messages) | ✅ 使用 PostgresSaver |
| **Store** | 长期记忆，跨会话持久化 | ❌ 不使用，用 Mem0 替代 |
| **Mem0** | 长期记忆 (factual + episodic + semantic) | ✅ 由 MemoryMiddleware 管理 |

### 5.2 消息裁剪机制

LangGraph checkpointer 会自动加载完整的对话历史到 `state["messages"]`，但当对话过长时需要裁剪。

**裁剪方式**：使用 LangChain Core 的 `trim_messages` 函数

```python
from langchain_core.messages.utils import trim_messages, count_tokens_approximately

def call_model(state):
    # 在调用模型前裁剪消息
    messages = trim_messages(
        state["messages"],
        strategy="last",                          # 保留最近的消息
        token_counter=count_tokens_approximately,
        max_tokens=8000,                          # 最大 token 数
        start_on="human",                         # 从 human 消息开始
        end_on=("human", "tool"),                 # 在 human/tool 消息结束
    )
    response = model.invoke(messages)
    return {"messages": [response]}
```

**Focus 中间件的职责**：
- 不加载历史（checkpointer 已加载）
- 在 before_model 时**先总结、再裁剪**（医疗场景关键信息不可丢失）
- 裁剪后的 messages 更新到 state，供后续节点使用

### 5.3 总结-裁剪策略（Summarize-then-Trim）

**为什么医疗场景必须先总结再裁剪？**

直接 `trim_messages` 会丢弃旧消息中的关键医疗信息（既往症状、已排除证型、用药反馈）。医疗对话中，早期对话可能包含对当前诊断至关重要的信息。因此主图的 Focus 中间件采用**总结-裁剪**两步策略：

```
messages 超过阈值
     │
     ▼
┌────────────────┐
│ Step 1: 总结    │  将超出窗口的旧消息摘要为一条 SystemMessage
│ (summarize)    │  保留关键医疗信息：症状、证型、用药、体征
└───────┬────────┘
        │
        ▼
┌────────────────┐
│ Step 2: 裁剪    │  trim_messages 移除已被总结的原始消息
│ (trim)         │  保留：摘要 SystemMessage + 最近 N 轮对话
└───────┬────────┘
        │
        ▼
  更新 state["messages"]
```

**实现方式**：在 FocusContextMiddleware.before_model 中集成

```python
from langchain_core.messages import SystemMessage
from langchain_core.messages.utils import trim_messages, count_tokens_approximately

async def _summarize_then_trim(self, messages, llm):
    """先总结再裁剪 - 医疗场景专用"""
    current_tokens = count_tokens_approximately(messages)
    if current_tokens <= self.config.max_tokens:
        return messages  # 未超阈值，不处理

    # Step 1: 提取需要被裁剪的旧消息，生成摘要
    keep_recent = self.config.keep_recent_messages  # 保留最近 N 条
    old_messages = messages[:-keep_recent] if keep_recent < len(messages) else []

    if old_messages:
        summary = await self._generate_medical_summary(old_messages, llm)
        summary_msg = SystemMessage(content=f"[对话历史摘要]\n{summary}")

        # Step 2: 用摘要替换旧消息，再 trim 保险
        new_messages = [summary_msg] + messages[-keep_recent:]
        return trim_messages(
            new_messages,
            strategy="last",
            token_counter=count_tokens_approximately,
            max_tokens=self.config.max_tokens,
            start_on="human",
        )

    return messages
```

**摘要 prompt 要求**（医疗场景特化）：
- 保留所有已提及的症状和体征
- 保留已确认/排除的证型
- 保留用药史和过敏反馈
- 保留关键时间线（发病时间、病程变化）
- 不保留寒暄和重复确认

## 6. 各层级中间件详细设计

### 6.1 L0 主图中间件

#### 设计原则

- 主图只做意图识别 + 路由, **不调用业务工具, 不执行推理诊断**
- 中间件负责加载全局上下文, 供路由器和所有子图使用
- Focus 使用 **总结-裁剪** 策略管理对话历史（先总结再裁剪，保留医疗关键信息）, **禁用工具裁剪**

#### 中间件链

```
优先级   中间件                    before_model              after_model
─────────────────────────────────────────────────────────────────────────
P0      GuardrailsMiddleware      安全检查(可拦截)           无
P5      FocusContextMiddleware    对话历史压缩(轻量)         无(路由快速,不触发)
P10     MemoryMiddleware          加载 factual+episodic      保存情景记忆到Mem0
                                  +semantic
P30     LearningMiddleware        加载 thread_learning       记录交互结果
                                  +cross_thread_learning     触发反思(仅错误时)
P40     PIIMiddleware             无                         输出脱敏
```

#### before_model 注入的完整上下文结构

**重要说明**: 这些上下文由中间件加载到 state 中，但**不会传递给意图识别器**。意图识别只使用用户查询和对话历史，保持简洁以避免干扰。这些上下文会在路由后传递给具体的业务处理节点（子图）使用。

### 5.2 L1 养生子图中间件

#### 设计原则

- 养生子图相对轻量, 不涉及复杂工具调用
- 继承主图已加载的 memory_context, **不重复检索**
- 需要独立的 Focus 压缩(养生回复可能较长)
- 学习经验以"有效养生策略"为主

#### 中间件链

```
优先级   中间件                    before_model              after_model
```

**注意**: 无 MemoryMiddleware (继承主图), 无 GuardrailsMiddleware (主图已检查), 无 PIIMiddleware (主图统一处理)

### 5.3 L1 诊断子图中间件

#### 设计原则

- 诊断子图是最复杂的子图, 包含信息收集循环、复杂度评估、三种辨证策略
- **需要完整的 Focus Sawtooth 压缩** (多轮收集产生大量消息)
- **需要工具输出裁剪** (moderate/complex 调用知识图谱、向量检索等)
- 继承主图 memory_context, 但需要独立的学习经验(工具选择、诊断纠错)
- 学习经验需要包含成功诊断案例(Procedural Memory)

#### 中间件链

```
优先级   中间件                    before_model              after_model
─────────────────────────────────────────────────────────────────────────
P5      FocusContextMiddleware    激进压缩+工具裁剪          阶段性压缩(complete_phase)
P30     LearningMiddleware        加载工具选择经验           记录诊断结果+工具调用
                                  +成功案例+鉴别规则         保存成功案例到数据库
```

#### before_model 注入

```python
{
    # === 继承自主图 (不重复注入) ===
    # state["memory_context"]  -> 包含 factual + episodic + semantic

    # === P5: FocusContextMiddleware (诊断专用, 完整版) ===
```

### 5.4 L2 DeepSearch Agent 中间件

#### 设计原则

- DeepSearch Agent 是诊断子图中 complex_diagnosis 路径的内部 Agent
- 使用 create_agent / create_deep_agent API, 拥有自己的中间件体系
- **工具调用极其密集** (KG + 向量 + 古籍 + Web + 5个专家子Agent)
- 需要最激进的上下文管理策略
- 继承诊断子图传入的 state (包含 memory_context + 已收集的症状信息)

#### 中间件链 (DeepSearch Agent 内部)

```

─────────────────────────────────────────────────────────────

P3      SummarizationMiddleware       上下文摘要 (DeepSearch 内置)
```

## 8. 设计决策记录 (ADR)

### ADR-01: 为什么子图不重复检索 Memory?

**决策**: 子图继承主图加载的 `memory_context`, 不独立检索 Mem0。

**原因**:
1. Factual (用户画像) 是稳定数据, 不会因进入子图而改变
2. Episodic (历史情景) 已在主图按 query 语义检索, 子图无需重复
3. Semantic (知识图谱) 是通用知识, 全局可用
4. 减少 Mem0 API 调用次数, 降低延迟和成本

**例外**: 如果未来子图需要"子图专属的记忆检索"(例如诊断子图需要检索特定证型的历史案例), 可以在子图的 MemoryMiddleware 中添加增量检索。

### ADR-02: 为什么成功案例既存为记忆又存为经验?

**决策**: 双重存储, 不构成冗余。

**原因**:
1. **情景记忆 (Mem0)**: 面向用户 → "你上次诊断了什么"
2. **程序记忆 (DB)**: 面向 Agent → "遇到这种情况用什么工具序列"
3. 存储内容不同: 记忆存摘要, 经验存工具调用序列和策略
4. 检索方式不同: 记忆用语义检索, 经验用规则匹配

### ADR-03: 为什么 DeepSearch Agent 不使用自定义中间件?

**决策**: DeepSearch Agent 使用 create_agent/create_deep_agent 内置的中间件体系。

**原因**:
1. DeepSearch Agent 有自己的 SummarizationMiddleware, 针对深度搜索优化
2. TodoListMiddleware + SubAgentMiddleware 是 DeepSearch 特有的能力
3. 避免两套中间件体系冲突
4. DeepSearch 的上下文由诊断子图通过 state 传入, 无需独立加载

### ADR-04: Focus 压缩发生在 before_model

**决策**: 所有层级的 Focus 压缩都在 `before_model` 阶段执行。

**原因**:
1. 压缩的目的是减少当前 LLM 调用的 token 数量
2. `before_model` 压缩直接降低当前调用的成本和延迟
3. `after_model` 压缩只能影响下一次调用, 当前调用已产生成本
4. 符合 Focus Sawtooth (arXiv:2601.07190) 论文的设计

### ADR-05: 主图 Focus 禁用工具裁剪

**决策**: 主图 FocusContextMiddleware 设置 `max_tokens_per_tool=0`。

**原因**:
1. 主图不调用任何业务工具
2. 主图只做意图识别 (轻量 LLM 分类调用)
3. 工具裁剪逻辑对主图是无效计算
4. 工具调用和推理诊断都在子图发生, 由子图自己管理

### ADR-06: 意图识别不使用 enriched_context

**决策**: 意图识别器只接收用户查询和对话历史，不使用 memory_context/focus_context/learning_context。

**原因**:
1. 意图识别是分类任务，需要保持输入简洁
2. 过多上下文会干扰分类器的判断准确性
3. 用户画像、历史情景、学习经验等应该在路由后传递给业务节点使用
4. 当前的意图识别方案已经是最佳实践，无需改动

**实现**: 中间件在 before_model 加载上下文到 state，但路由器不读取这些字段，只读取 messages。
